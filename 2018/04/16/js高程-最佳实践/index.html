js高程-最佳实践

发表于 2018-04-16 | 分类于 前端__js

　　这部分内容虽说是理论性的，但我觉得其重要性不亚于js的基础语法，我在读完这部分后知道了如何正确规范的书写自己的代码，好的代码不仅是可以让自己方便维护，重要的是能让别人更快的读懂，并开展维护及工作。
1.首先是代码的约定

  • 可读性：要想让代码可维护，它得是可读的，重要的是写注释。（1）每个函数和方法都应该包含一个注释，用来描述其作用或者参数的含义。（2）在大段代码前放置一个描述任务的注释。（3）如果用了独特的方式解决了某个问题，应注释如何解决，方便自己或其他开发者读懂。（4）缩进和注释可以带来更可读的代码，在未来则更容易维护
  • 变量和函数名：(1)变量名应为名词如car或person;(2)函数名应该以动词开始,如getName()。返回布尔值函数一般以is开头，如isEnable();(3)变量和函数都应使用合乎逻辑的名字，不用担心长度，后期部署时可以压缩。
  • 变量类型透明：js变量是松散类型，根据数据动态匹配类型，因此判断数据类型就落到的开发者身上（1）当定义一个变量后就通过初始化为该数据类型的值，如var found=false //布尔值。（2）用于指定类型的类型注释，如var found /Boolean/ =false　　

2.松散耦合过紧，难以维护

  • 解耦HTML/Javascript:常见的耦合是在HTML代码中嵌入script，或者是使用html事件处理程序，还有通过innerHTML插入带有html标签的文本。实践中，应该将结构和行为分开，否则出现错误是得判断其出现在js文件中还是html部分
  • 解耦css/javascript:应该减少或不直接用js来更改页面样式，如element.style.color=”red”,css部分就直接在css中查看或修改。当然现在常常是用js来更改样式，可以通过动态的更改规则名来修改，如element.className=”edit”,这样的话就不需要将样式写在js中了
  • 解耦应用逻辑/事件处理程序：在事件程序中处理应用逻辑会让调试更困难，如果出现错误，就得先判断是事件程序还是应用逻辑的问题。另外分离后的代码可以更好的复用，比如更容易的更改触发事件类型。几条原则：（1）勿将event对象传给其它方法；（2）任何可以在应用层面的动作都可以在不执行任何事件程序的情况下进行；（3）任何事件程序都应该处理事件，然后将处理转交给应用逻辑。　　

　
3.编程实践

  • 尊重对象所有权：在ES5中通过引入防篡改对象来解决。如果不负责创建或维护某个对象，那么就不要为实例或原型添加属性和方法，也不要重定义已存在的方法，否则会引起意料之外的错误，更不能对原生类型和对象进行这种操作。
  • 避免全局量：最多创建一个全局变量，让其他对象和函数存在其中。单一的全局量的延伸便是命名空间的概念
  • 避免与null进行比较：验证数据类型时，可以直接用typeof或instanceof,这样可以阻止非法值，不必用null　　
  • 使用常量：将数据从应用逻辑分离出来的思想，可以在不冒引入错误的风险的同时，就改变数据　　

4.性能

  • 注意作用域：只要能减少花费在作用域链上的时间，就能增加脚本的整体性能。（1）避免全局查找，可以将多次用到的全局对象存储为局部变量。（2）避免使用with语句，会增加作用域链的长度
  • 选择正确方法：（1）避免不必要的属性查找、算法值从简到繁：O(1)常数 < O(log n)对数 < O(n)线性 < O(n*n)平方。（2）优化查询：减值迭代、简化终止条件、简化循环体、使用后测试循环，如for循环和while循环是前测试循环，do-while是后循环，可以避免最初终止条件的计算，因此运行更快
  • 多个变量声明可以使用一个var,之间用逗号隔开　　

5.优化dom交互

  • 使用循环将新建元素添加到文档中时，可以先将这些元素循环后统一放到一个容器中，然后再将这个容器添加到文档中。
  • 使用事件代理　　　

6.部署：将代码分离成多个文件只是为了提高可维护性，并非为了部署。要进行部署的时候，需要将这些源代码合并为一个或几个归并文件。

ps:分享一个小提示，当我发布这篇博客时，遇到了一个问题，就是发布上去的文章结构错乱，本来用~~~包含的代码块没有正常包含，却包含住了下面的内容，研究了半个小时，偶然才发现该代码块结束的~~~后面被我多敲了几个空格，使得该~~~无效了，就和下一个代码块的开头~~~合成了一个代码块，以后得注意这种问题
